# Plan: Integrate MyPayroll into IHRP (v2 ‚Äî Final)

## Context

Bring all MyPayroll features (YTD income dashboard, tax breakdown, annual projections, multi-year trends, consultant breakdown) into the IHRP Laravel app as a unified payroll module. Admin uploads `.xlsx` payroll files; data is parsed, stored in MySQL, and rendered via a Payroll dashboard with Chart.js (CDN).

### What's New in v2

The original plan assumed a single-owner payroll view. This revision expands the data model to support **4 account managers** (Raf + 3 others), each with their own uploaded payroll data. 3 AMs have existing payroll history; 1 AM is brand new with no placements or payroll data yet.

- Every payroll record is scoped to an **owner** (the AM whose payroll file was uploaded)
- The DB is pre-populated and kept current through regular uploads from all active AMs
- Admins see aggregate and per-AM views; AMs see only their own payroll data
- Consultant breakdown is scoped per-AM via placement linkage
- New AMs with no data see a clean empty state (not broken charts or division-by-zero errors)

**Team composition:**

| AM | Status | Initial Data |
|----|--------|-------------|
| Raf | Active, multi-year history | Full historical XLSX upload |
| AM #2 | Active, has placements | Historical XLSX upload |
| AM #3 | Active, has placements | Historical XLSX upload |
| AM #4 | New hire, no placements yet | No upload; empty state until first payroll file |

**Access model:**

| Role | Payroll Dashboard | Upload | Goal Setting | Consultant Breakdown |
|------|------------------|--------|--------------|---------------------|
| `admin` | All 4 AMs (aggregate + per-AM toggle) | Yes (any AM's file) | Yes (per AM) | All consultants |
| `account_manager` | Own data only | No (admin handles all uploads) | View only | Own placements only |

> **Why admin-only uploads:** Payroll XLSX files are generated by an external payroll system first. Admin receives the output and uploads into IHRP. AMs never need to touch the upload flow.

## New Files to Create (nothing existing is changed)

### 1. Migrations (run in order)

#### `create_payroll_uploads_table`
```
id, user_id (FK ‚Üí users, the AM who owns this file), original_filename,
stored_path, stop_name (VARCHAR, the stop-row name used during parse),
record_count, warnings (JSON), created_at, updated_at
```

> `stop_name` is stored here so any upload can be audited: you can always reproduce exactly what was parsed and why rows were truncated.

#### `create_payroll_records_table`
One row per check date **per owner**.
```
id, user_id (FK ‚Üí users, the AM/owner),
check_date (DATE), gross_pay, net_pay, federal_tax, state_tax,
social_security, medicare, retirement_401k, health_insurance,
other_deductions, commission_subtotal, salary_subtotal,
created_at, updated_at

UNIQUE(user_id, check_date)   ‚Üê composite unique, not just check_date
```
All money columns: `DECIMAL(12,4)`.

#### `create_payroll_consultant_entries_table`
Per consultant per year **per owner**.
```
id, user_id (FK ‚Üí users, the AM/owner),
consultant_name (VARCHAR), year (SMALLINT),
revenue, cost, margin, pct_of_total,
consultant_id (FK ‚Üí consultants, nullable),
created_at, updated_at

UNIQUE(user_id, consultant_name, year)
```

#### `create_payroll_consultant_mappings_table`
Persistent name-to-ID mappings. Populated once, reused on every subsequent upload. Admin can fix mismatches via UI.
```
id, raw_name (VARCHAR, the name as it appears in the XLSX),
consultant_id (FK ‚Üí consultants), user_id (FK ‚Üí users, the AM context),
created_by (FK ‚Üí users), created_at, updated_at

UNIQUE(raw_name, user_id)
```

#### `create_payroll_goals_table`
One row per year **per owner**.
```
id, user_id (FK ‚Üí users), year (SMALLINT), goal_amount DECIMAL(12,4),
created_at, updated_at

UNIQUE(user_id, year)
```

### 3. PayrollParseService

File: `app/Services/PayrollParseService.php`

**Responsibility:** Raw XLSX extraction only. Takes a file and a stop name, returns normalized arrays. No DB writes, no aggregation.

```php
public function parse(UploadedFile $file, string $stopName): PayrollParseResult
// Returns DTO: { ownerName, records[], consultantsByYear[], warnings[] }

private function parseSummarySheet(Spreadsheet $wb, string $stopName): array
private function parseConsultantSheets(Spreadsheet $wb): array
private function isPeriodSheet(string $name): bool
private function getSheetYear(Worksheet $ws): ?int
private function parsePayCalc(Worksheet $ws, string $stopName): array
```

> **Why `$stopName` is a parameter, not a config key:** Each AM's payroll file stops at a different row (the row that starts with that AM's name signals end-of-data). A global setting can only hold one value, which fails once you have multiple AMs. Passing `$stopName` per parse call makes the service a pure function of its inputs ‚Äî same file + same stop name always produces the same result, with no hidden state. It also makes the service trivial to unit test.

**File validation (before parsing):**
- MIME type check: `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`
- File size: reject > 50 MB at controller level (`max:51200` validation rule)
- Sheet structure: verify expected summary sheet exists; return 422 with descriptive error if not
- Header detection: verify required columns exist before attempting row extraction

**Critical porting notes:**
- `"Social Security "` has a trailing space ‚Äî normalize via `trim()` during header detection
- Date cells: use `ExcelDate::isDateTime($cell)`; fallback to `DateTime::createFromFormat('m/d/Y', $value)`
- Pandas `groupby` ‚Üí PHP associative array keyed by ISO date string, summing numeric cols
- Pandas `resample("MS")` ‚Üí group by `date('Y-m', strtotime($date))`, first-of-month as key
- `"Commission...Subtotal"` tier detection: also handle typo `"Subttal"` (present in original)
- Stop condition: row starts with `$stopName` (case-sensitive, `str_starts_with`) ‚Äî stop reading; do not include that row
- Set `ini_set('memory_limit', '256M')` at parse start

### 4. PayrollDataService

File: `app/Services/PayrollDataService.php`

Handles all aggregation, projection math, and goal logic. Controller stays thin.

```php
public function getYears(int $userId): array
public function getSummary(int $userId, int $year): array
public function getMonthly(int $userId, int $year): array
public function getAnnualTotals(int $userId): array
public function getConsultants(int $userId, int $year): array
public function getProjection(int $userId, int $year): array
```

**Projection logic (explicit):**
- Method: linear extrapolation ‚Äî `(YTD net √∑ periods elapsed) √ó 26`
- Periods elapsed = count of `payroll_records` for that user+year
- **Suppression rule:** if periods elapsed < 4, return `null` with a flag `projectionSuppressed: true` and message "Too early in year to project reliably"
- Prior-year comparison: include last year's final total alongside projection

**Aggregate (admin) methods:**
```php
public function getAggregateSummary(int $year): array   // sum across all AMs
public function getPerAmBreakdown(int $year): array     // per-AM totals for admin comparison
```

> **`getPerAmBreakdown` implementation note:** Build the AM list by querying `User::where('role', 'account_manager')->orderBy('name')->get()`. Never hard-code the 4 AMs ‚Äî this query automatically includes future hires and excludes anyone whose role changes. Left-join payroll_records for the given year so AMs with no data appear with `$0` across all fields, never omitted.

**Null-safety for new/empty AMs:**
- `getPerAmBreakdown` always returns all `account_manager` users regardless of data. AMs with no records show `$0` across all fields ‚Äî never omitted from the list.
- `getAggregateSummary` sums only non-null values; an AM with zero records contributes nothing (not `null` that breaks arithmetic).
- Percentage calculations (e.g., "AM share of total revenue") must guard against division-by-zero when total is `$0`.
- `getProjection` returns `{ projectionSuppressed: true, reason: 'no_data' }` when record count is 0 (distinct from the < 4 periods suppression which returns `reason: 'too_early'`).

### 5. PayrollController

File: `app/Http/Controllers/PayrollController.php`

| Method | Auth | Notes |
|--------|------|-------|
| `index()` | `account_manager` | Returns `view('payroll.index')` ‚Äî no data, all via JS |
| `upload(Request)` | `admin` | Parse ‚Üí resolve mappings ‚Üí DB::transaction ‚Üí auditLog ‚Üí JSON. Admin must specify target AM via `user_id` and stop name via `stop_name`. |
| `apiDashboard(Request)` | `account_manager` | **Consolidated endpoint**: returns `{years, summary, monthly, annualTotals, goal, projection}` for one user+year |
| `apiConsultants(Request)` | `account_manager` | Separate (drawer-triggered): AM-scoped |
| `apiAggregate(Request)` | `admin` | Admin only: cross-AM summary |
| `apiGoalSet(Request)` | `admin` | `updateOrCreate` on `(user_id, year)`. Validate `user_id` references an AM. |
| `apiMappings(Request)` | `admin` | GET: unresolved names; PUT: set consultant_id for a raw_name |

**Why consolidated `/api/dashboard`:**
The original plan had 7 separate GET endpoints. Every page load fired 5-6 requests before anything rendered. The dashboard endpoint returns everything needed for initial render in one payload (~5-10 KB). Consultant data stays separate since it's behind a drawer click.

**Upload flow (admin-only):**
```
1. Validate file (MIME, size, structure)
2. Validate target user_id is present and references a valid AM (role = account_manager);
   return 422 if missing or if user is not an AM
3. Validate stop_name is present and non-empty; return 422 if missing
4. Parse via PayrollParseService::parse($file, $stopName) ‚Üí PayrollParseResult
5. Resolve consultant names:
   a. Check payroll_consultant_mappings for existing raw_name ‚Üí consultant_id
   b. For unmapped names: attempt case-insensitive match against consultants table
   c. Auto-create mapping for successful matches
   d. Log unresolved names in warnings[] (surface in upload response)
6. DB::transaction {
     Upsert payroll_records (keyed on user_id + check_date)
     Determine affected_years = array_unique(array_column($result->consultantsByYear, 'year'))
     Delete payroll_consultant_entries WHERE user_id = $targetUserId AND year IN ($affected_years)
     Insert new consultant_entries for those years
     Create PayrollUpload record (including stop_name)
   }
7. AuditLog: user_id, target_am_id, filename, stop_name, record_count,
             years_affected[], unresolved_names[], warnings[]
8. Return JSON: { success, recordCount, yearsAffected, unresolvedNames[], warnings[] }
```

> **"Affected years" definition:** The years present in the uploaded file's consultant sheets ‚Äî derived from `$result->consultantsByYear`, not from querying the existing DB. This ensures a partial-year re-upload only touches consultant data for years it actually contains, leaving prior years untouched.

**Owner scoping (read endpoints):**
```php
private function getOwnerId(Request $request): int
{
    if (Auth::user()->isAdmin() && $request->has('user_id')) {
        $user = User::findOrFail($request->integer('user_id'));
        abort_if($user->role !== 'account_manager', 422, 'Target user must be an account manager');
        return $user->id;
    }
    return Auth::id();
}
```

All read API methods use this. AMs always resolve to `Auth::id()`. Admins pass `?user_id=X` to view a specific AM, or omit for aggregate. Passing an admin's own `user_id` is rejected with 422 ‚Äî not silently accepted.

## Data Lifecycle

### Initial Population
1. Admin uploads Raf's historical payroll XLSX ‚Äî enters `stop_name = "Rafael Zobel"` (or whatever form his name takes in the file)
2. Admin uploads AM #2's payroll XLSX ‚Äî enters AM #2's stop name
3. Admin uploads AM #3's payroll XLSX ‚Äî enters AM #3's stop name
4. AM #4 has no file to upload ‚Äî dashboard shows empty state automatically
5. Admin resolves any unmatched consultant names via mapping UI
6. Admin can optionally set goals for all 4 AMs (including AM #4 as a target before first payroll)
7. Dashboard is immediately functional for all 4 AMs

### Empty State (AM #4 and any future new hires)
When an AM has zero `payroll_records`:
- KPI cards show `$0` / `‚Äî` (not broken or blank)
- Charts render with empty datasets (Chart.js handles this gracefully with `data: []`)
- Projection shows "No payroll data yet" instead of a number
- Goal tracker shows 0% progress if a goal is set, or "No goal set" if not
- Consultant drawer shows "No consultant data for this period"
- Upload button not visible to AMs (admin-only); empty state shows "No payroll data available yet"
- The AM is still listed in admin's per-AM breakdown with `$0` across the board

### Ongoing
- Each pay period, admin uploads the latest payroll XLSX for each AM, re-entering the stop name (or the modal could pre-fill it from the most recent `payroll_uploads.stop_name` for that AM)
- Upsert ensures existing records update, new records insert
- Consultant mappings persist ‚Äî no re-matching needed after first resolution
- Soft-delete behavior: records NOT in the new file are **preserved** (upsert only touches rows present in the upload). This is intentional ‚Äî a partial-year upload doesn't wipe earlier data.

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| PhpSpreadsheet date cell detection | HIGH | Fallback to string date parsing; log warning; unit test coverage |
| Memory on large XLSX | MEDIUM | `ini_set('256M')`; enforce 50 MB upload limit at controller |
| Consultant name mismatches | MEDIUM | `payroll_consultant_mappings` table with admin UI; auto-match on first pass, manual fix for outliers |
| Stop-name entry error (admin types wrong name) | LOW | Wrong stop_name ‚Üí wrong record count surfaced in upload response; re-upload corrects it (upsert). `payroll_uploads.stop_name` lets admin see what was used. |
| Re-upload partial-year coverage | LOW | Intentional: only affected check_dates are upserted; document this behavior |
| Concurrent upload during read | LOW | Consultant delete+reinsert wrapped in same DB::transaction as record upsert |
| AM payroll file format differences | MEDIUM | Validate sheet structure before parsing; return descriptive 422 if format doesn't match expected layout |
